首先docker启动一个rabbitMQ服务

rabbitmq:
container_name: "${APP_NAME}-rabbitmq"
image: rabbitmq:4.0-management
restart: always
environment:
RABBITMQ_DEFAULT_USER: Qh
RABBITMQ_DEFAULT_PASS: Qh
volumes:
- volume-rabbitmq-data:/var/lib/rabbitmq
ports:
- "${RABBITMQ_PORT}:5672"
- "15672:15672"
networks:
- network-internal

环境变量

.env
MESSENGER_TRANSPORT_DSN=amqp://Qh:Qh@rabbitmq:5672/%2f
MESSENGER_FAILURE_TRANSPORT_DSN=doctrine://default?queue_name=failed

依赖包

comspoer包:
"symfony/amqp-messenger": "7.2.*",

messenger配置文件

messenger.yaml文件

framework:
messenger:
failure_transport: failed
        transports:
            # 邮件消息队列 transport
            Qh_mail_transport:
                dsn: '%env(MESSENGER_TRANSPORT_DSN)%'
                failure_transport: failed
                retry_strategy:
                    max_retries: 3
                    multiplier: 2
                options:
                    exchange:
                        name: 'Qh_exchange'          # 交换机名字
                        type: 'direct'
                        default_publish_routing_key: 'mail_routing_key'
                    queues:
                        Qh_mail_queue:                # 队列名字, queues 下可以有多个队列名字，然后通过 binding_keys 和不同的交换机关联
                            binding_keys: ['mail_routing_key']

            # 失败队列 transport
            failed:
                dsn: '%env(MESSENGER_FAILURE_TRANSPORT_DSN)%'
                options:
                    queue_name: failed_queue             # 失败队列名字

        routing:
            App\Message\QhMail: Qh_mail_transport #QhMail 消息会发送到 Qh_mail_transport，Qh_mail_transport 绑定队列名 Qh_mail_queue，所以启动 messenger:consume Qh_mail_transport 就会消费到队列 Qh_mail_queue 的消息

作用机制:

  • 1.消息对象 QhMail 被 dispatch 到 Qh_mail_transport
  • 2.Transport 会把消息发到 Qh_exchange 交换机
  • 3.交换机根据 routing key mail_routing_key 将消息路由到 Qh_mail_queue
  • 4.Worker 消费 Qh_mail_queue
  • 5.如果消息处理失败，发送到 failed_queue

消费者, 消费 Qh_mail_transport 对应的队列, 每个 transport 独立 worker

php bin/console messenger:consume Qh_mail_transport

为什么推荐 每个 transport 独立 worker

  • 假设有多个消息类型：
邮件消息 → transport: Qh_mail_transport
短信消息 → transport: Qh_sms_transport
推送通知 → transport: Qh_push_transport
  • 如果一个 worker 消费 所有 transport：
邮件堆积时，可能阻塞短信或推送队列的消息
不同类型消息的处理速度不同，混合在一个 worker 不好控制
  • 独立 worker 优点：
每个消息类型有自己的消费进程
可以单独扩容，邮件积压时多开几个邮件 worker，不影响短信或推送

每个 transport 只能对应一个交换机，但是RabbitMQ其实是可以定义多个交换机的，为了实现这一点，可以定义多个 transport 然后关联不同的交换机，如下:

framework:
messenger:
transports:
mail_transport:
dsn: '%env(MESSENGER_TRANSPORT_DSN)%'
options:
exchange:
name: 'Qh_mail_exchange'
type: 'direct'
default_publish_routing_key: 'mail_routing_key'
queues:
Qh_mail_queue:
binding_keys: ['mail_routing_key']

            sms_transport:
                dsn: '%env(MESSENGER_TRANSPORT_DSN)%'
                options:
                    exchange:
                        name: 'Qh_sms_exchange'
                        type: 'direct'
                        default_publish_routing_key: 'sms_routing_key'
                    queues:
                        Qh_sms_queue:
                            binding_keys: ['sms_routing_key']

        routing:
            App\Message\MailMessage: mail_transport
            App\Message\SmsMessage: sms_transport

然后:

php bin/console messenger:consume mail_transport
php bin/console messenger:consume sms_transport
分别启动两个 worker.

手动生成需要的交换机等配置信息:

php bin/console messenger:setup-transports
这个命令会自动声明 RabbitMQ 中所有必要的 exchange、queue、binding。常用于部署时初始化。

拓展:

transport name 和queue name的区别

🧩 一、简单结论先说

概念示例所属系统含义
transport namemail_transportSymfony Messenger 配置层Symfony 应用中定义的一种“消息传输通道”——比如 RabbitMQ、Redis、Doctrine 等。
queue nameQh_mail_queueRabbitMQ 实际层RabbitMQ 内部存放消息的队列名称。

👉 换句话说：

  • transport name 是 Symfony 层的别名（逻辑名称）
  • queue name 是消息中间件（RabbitMQ）层的真实队列名称

⚙️ 二、在配置中的位置与作用

假设你的 messenger.yaml 里写：

framework:
    messenger:
        transports:
            mail_transport:
                dsn: '%env(MESSENGER_TRANSPORT_DSN)%'
                options:
                    exchange:
                        name: 'Qh_exchange'
                        type: 'direct'
                        default_publish_routing_key: 'Qh_mail'
                    queues:
                        Qh_mail_queue:
                            binding_keys: ['Qh_mail']

        routing:
            App\Message\QhMail: mail_transport

那么解释如下 👇

元素说明
mail_transportSymfony Messenger 定义的“传输通道名称”，相当于“连接配置”或“逻辑标识”。
dsn指明该 transport 使用的连接方式，比如 amqp://user:pass@rabbitmq:5672/%2f。
exchange.nameRabbitMQ 中的交换机名。
queues.Qh_mail_queue在 RabbitMQ 上实际创建的队列。
binding_keys队列绑定到交换机时的路由键（routing key）。
routing:表示当 dispatch 某个消息类时，使用哪个 transport（比如 mail_transport）。

🧠 三、执行时的对应关系

假设代码中这样写：

$bus->dispatch(new QhMail('user@example.com', 'Hello'));

流程是这样的：

  1. Messenger 查找 routing：

    App\Message\QhMail: mail_transport

    → 发现应该发到 mail_transport

  2. mail_transport 配置指向 RabbitMQ DSN → Messenger 创建连接到 RabbitMQ 服务器

  3. Messenger 使用配置的 exchange 和 default_publish_routing_key

    exchange:
    name: Qh_exchange
    default_publish_routing_key: Qh_mail

    把消息发送到交换机。

  4. RabbitMQ 将消息根据 binding_keys 规则投递到队列：

    Qh_mail_queue

  5. 消费者执行：

    php bin/console messenger:consume mail_transport

    → Messenger 通过 transport 名找到配置的 RabbitMQ 队列 → 开始监听 Qh_mail_queue

🧩 四、类比帮助理解

可以类比成一个“邮政系统”：

层级对象比喻
Messenger transport邮局（比如“上海邮局”）负责投递消息到某个系统
Exchange邮件分拣中心根据地址把信件分派到正确的邮袋
Queue邮袋存放待派送的邮件
Routing key邮件标签指明信件该放进哪个邮袋

✅ 五、推荐命名规范（避免混乱）

建议配置时明确区分逻辑层与物理层名称，比如：

transports:
    Qh_mail_transport:
        dsn: '%env(MESSENGER_TRANSPORT_DSN)%'
        options:
            exchange:
                name: 'Qh_exchange'
                type: 'direct'
                default_publish_routing_key: 'mail_key'
            queues:
                Qh_mail_queue:
                    binding_keys: ['mail_key']
  • transport name: Qh_mail_transport（Symfony逻辑名）
  • exchange name: Qh_exchange（RabbitMQ交换机）
  • queue name: Qh_mail_queue（RabbitMQ队列）

这样看起来就很清晰，不容易搞混。
