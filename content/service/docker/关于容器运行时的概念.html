📜 第一部分

关于容器运行时的概念

一、什么是「运行时」(Runtime)

核心定义

运行时 = 把"描述"变成"正在运行的进程"的那一层

更具体一点：

运行时负责：

  • 创建 Linux 进程
  • 设置 namespace(pid / net / mount / user…)
  • 设置 cgroup(CPU / memory / io)
  • 挂载 rootfs
  • exec() 你的程序

📌 只要真正做了这些事，就叫运行时

运行时 ≠ 构建工具

动作是运行时吗
docker build❌
docker push❌
docker pull❌
docker run(CLI)❌(只是入口)
runc✅
containerd(对 K8s)✅(高层)

二、Docker 里的「运行时」是怎么体现的？

Docker 架构(简化)

docker CLI
   ↓
Docker Engine
   ↓
containerd
   ↓
runc   ← 真正的运行时
   ↓
Linux 内核

Docker 中的"运行时分层"

1️⃣ Docker Engine(不是运行时)

  • 提供 API(docker ps / logs)
  • 管理镜像、网络、volume
  • 不直接创建 Linux 进程

❌ 不是运行时

2️⃣ containerd(高层运行时)

  • 管理容器生命周期
  • 管理镜像
  • 调用 runc

⚠️ 在 Docker 里是"内部组件"

3️⃣ runc(低层运行时 / 真·执行者)

  • 实现 OCI Runtime Spec
  • 直接调用 Linux 内核
  • 创建容器进程

✅ Docker 里的真正运行时

Docker 下运行流程(一句话)

docker run
 → Docker Engine
 → containerd
 → runc
 → Linux 进程

三、Kubernetes 里的「运行时」是怎么体现的？

Kubernetes 的设计原则

K8s 不关心你用什么容器技术，只要你实现接口

所以它定义了：

CRI(Container Runtime Interface)

K8s 架构(标准形态)

kubelet
   ↓
CRI
   ↓
containerd / CRI-O   ← K8s 的运行时
   ↓
runc / crun          ← OCI Runtime
   ↓
Linux 内核

Kubernetes 中的"运行时"两层

1️⃣ CRI Runtime(K8s 语境里的"运行时")

  • containerd
  • CRI-O

职责：

  • 实现 CRI 接口
  • 拉镜像
  • 创建 / 管理 Pod 中的容器
  • 调用 OCI Runtime

📌 K8s 官方说的"runtime"通常指这一层

2️⃣ OCI Runtime(真正执行层)

  • runc
  • crun
  • kata-runtime

职责：

  • 创建 namespace / cgroup
  • exec 容器进程

📌 和 Docker 完全一致

Kubernetes 下运行流程

Pod YAML
   ↓
kubelet
   ↓
CRI (gRPC)
   ↓
containerd / CRI-O
   ↓
OCI Runtime (runc)
   ↓
Linux 进程

四、OCI 在 Docker 和 K8s 中的位置

OCI 是什么？

OCI = 规范，不是程序

它规定两件核心事情：

1️⃣ OCI Image Spec(镜像规范)

Docker / K8s 通用

  • Docker build → OCI Image
  • containerd pull → OCI Image
  • K8s image: xxx → OCI Image

📌 "Docker 镜像"本质 = OCI 镜像

2️⃣ OCI Runtime Spec(运行规范)

Docker / K8s 通用

  • runc / crun 都实现它
  • 定义 config.json
  • 定义容器启动方式

📌 所有运行时的"宪法"

OCI 在流程中的体现

构建阶段：
Docker build
 → OCI Image Spec

运行阶段：
containerd / CRI-O
 → OCI Runtime Spec
 → runc

五、CRI 在哪里？只属于 Kubernetes

CRI 是什么？

CRI = Kubernetes 定义的"运行时接口"

  • gRPC 接口
  • 只存在于 K8s
  • Docker 没有 CRI

CRI 解决的问题

kubelet 不想适配每一种容器实现

所以：

  • kubelet ↔ CRI
  • containerd / CRI-O 实现 CRI

CRI 在流程中的体现

kubelet
   ↓(CRI)
containerd / CRI-O

📌 CRI 把 K8s 和具体容器技术解耦

六、一张"终极对照表"(建议保存)

概念DockerKubernetes
镜像规范OCI ImageOCI Image
运行时接口❌CRI
高层运行时containerd(内部)containerd / CRI-O
低层运行时runcrunc / crun
真正 execruncrunc
Docker Engine必需❌ 不存在

七、最终一句话总结(可以背)

运行时 = 把 OCI 镜像变成 Linux 进程的系统

  • Docker：运行时藏在 Docker Engine 里面
  • Kubernetes：运行时是显式的(CRI → OCI Runtime)
  • OCI：统一"镜像格式"和"运行规则"
  • CRI：K8s 专用的"运行时接口"

📜 第二部分

举例说明: 同一个 nginx 镜像 对比 docker run 和 kubectl run 的真实执行路径

我们用 同一个 nginx 镜像，把
👉 docker run nginx
👉 kubectl run nginx
的真实执行路径逐层拆开，对照 运行时 / CRI / OCI。

一、先给结论

docker run → 直接走 Docker 自己的运行链路

kubectl run → 走 Kubernetes → CRI → containerd / CRI-O → OCI Runtime

镜像是同一个，运行路径完全不同。

二、docker run nginx：真实执行路径

你敲的命令

docker run nginx

实际发生的链路(精简但准确)

docker CLI
   ↓  REST API
Docker Engine (dockerd)
   ↓
containerd
   ↓
OCI Runtime (runc)
   ↓
Linux kernel
   ↓
nginx 进程

每一层干了什么？

1️⃣ docker CLI

  • 解析命令
  • 调用 Docker Engine API

2️⃣ Docker Engine(dockerd)

  • 创建容器元数据
  • 创建网络 / volume
  • 调 containerd

3️⃣ containerd

  • 拉 nginx 镜像(OCI Image)
  • 准备 rootfs
  • 生成 OCI config.json

4️⃣ runc(OCI Runtime)

  • 创建 namespace / cgroup
  • mount rootfs
  • exec nginx

📌 真正运行 nginx 的是 runc

5️⃣ Linux kernel

  • nginx 成为一个普通进程(被隔离)

你在系统里看到的结果

docker ps
docker logs
docker inspect

三、kubectl run nginx：真实执行路径

你敲的命令

kubectl run nginx --image=nginx

实际发生的链路(精简但准确)

kubectl
   ↓  REST
kube-apiserver
   ↓
etcd
   ↓
kubelet (node)
   ↓  CRI (gRPC)
containerd / CRI-O
   ↓
OCI Runtime (runc)
   ↓
Linux kernel
   ↓
nginx 进程

每一层干了什么？

1️⃣ kubectl

  • 把命令转换成 Pod YAML
  • 发给 kube-apiserver

2️⃣ kube-apiserver

  • 校验
  • 写入 etcd

3️⃣ kubelet(节点上)

  • watch 到新 Pod
  • 调用 CRI 接口

4️⃣ CRI Runtime(containerd / CRI-O)

  • 拉 nginx 镜像(OCI Image)
  • 创建 Pod Sandbox
  • 准备容器

5️⃣ runc(OCI Runtime)

  • 完全同 docker run
  • 创建 namespace / cgroup
  • exec nginx

📌 这里和 Docker 完全一致

你在系统里看到的结果

kubectl get pods
kubectl logs nginx
crictl ps

❌ docker ps 看不到

四、两条路径并排对照(重点)

对比点docker runkubectl run
镜像nginx(OCI)nginx(OCI)
命令入口docker CLIkubectl
调度❌✅
API Server❌✅
CRI❌✅
Docker Engine✅❌
containerd✅✅
runc✅✅
真正 execruncrunc

五、最容易误解的一点

Docker 并没有"跑容器"Kubernetes 也没有"跑容器"

真正跑容器的，永远是：

OCI Runtime(runc)

六、一个验证方法

在 K8s 节点上：

ps aux | grep nginx

你会看到：

  • docker run 的 nginx
  • kubectl run 的 nginx

它们在内核层面完全一样

七、最终一句话总结

同一个 nginx 镜像：

  • docker run 走 Docker Engine 链路
  • kubectl run 走 Kubernetes → CRI 链路

但在最后一跳，它们都由 runc 创建 Linux 进程，只是到达containerd的方式不同

📜第三部分

到达containerd的方式不同 详细分析

docker run 和 kubectl run 的核心区别： 是「到达 containerd 的方式不同」

而不是：

  • 镜像不同 ❌
  • runc 不同 ❌
  • Linux 内核不同 ❌

用一句"路径表达式"来对比

docker run

docker CLI
 → Docker Engine (dockerd)
 → containerd
 → runc
 → Linux process

kubectl run

kubectl
 → kube-apiserver
 → kubelet
 → CRI (gRPC)
 → containerd
 → runc
 → Linux process

📌 containerd 是共同点 📌 分叉发生在 containerd 之前

为什么这个区别"非常重要"？

因为 containerd 的"上游调用者"决定了行为模式。

一、谁在"指挥" containerd？

docker run

  • 指挥者：Docker Engine

  • containerd 是：

    • Docker 的内部组件
    • 只服务 Docker

  • containerd 接收的是 Docker 自己的 API 语义

kubectl run

  • 指挥者：kubelet

  • containerd 是：

    • Kubernetes 的 runtime
    • 通过 CRI 交互

  • containerd 接收的是 Kubernetes 的 Pod 语义

二、containerd 接收到的"指令语义"不同

Docker 语义

RunContainer(
  image=nginx,
  ports=80,
  network=bridge,
  restart=always
)

Kubernetes 语义(CRI)

RunPodSandbox()
CreateContainer()
StartContainer()

📌 containerd 只是"翻译执行者"

三、为什么 Kubernetes 不用 Docker Engine？

因为 Docker Engine：

  • ❌ 不支持 CRI
  • ❌ API 语义是"容器"，不是"Pod"
  • ❌ 多了一层状态机

所以 Kubernetes 选择：

直接对接 containerd / CRI-O

四、一个非常精准的工程判断标准(送你)

看 containerd 的"上游是谁"，就知道这是 Docker 还是 Kubernetes 的容器

五、你这句话可以升级成"标准表述"

你现在说的：

最主要的是到达 containerd 的方式不同

升级后可以这样说(技术交流很加分)👇

docker run 和 kubectl run 的本质区别在于： containerd 的上游调用者不同—— Docker Engine vs kubelet(通过 CRI)

六、最终一句话总结

containerd 是共同的执行核心， Docker 和 Kubernetes 的差异发生在 containerd 之上，而不是之下。

📜第四部分

用 containerd 的视角，展示它如何同时"服务 Docker 和 K8s"

"我叫 containerd，我不关心你是 Docker 还是 Kubernetes， 谁按我听得懂的接口跟我说话，我就帮谁把容器跑起来。"

一、containerd 是一个什么角色？

核心身份

containerd = 通用容器运行管理器

它的职责只有三件：

  1. 管理镜像(OCI Image)
  2. 管理容器生命周期
  3. 调用 OCI Runtime(runc)

二、containerd 如何"同时服务 Docker 和 K8s"？

关键点只有一个：

containerd 对外提供了"多种入口(frontend)"， 对内只有"一套执行逻辑"。

三、从 containerd 的"接口视角"看世界

containerd 的内部核心是统一的

Image Store
Snapshotter
Task Manager
Runtime (runc)

不管谁来，最后都会走这里 👆

四、Docker 是怎么"接入我"的？

Docker → containerd

Docker Engine
   ↓
containerd gRPC API
   ↓
containerd Core

特点

  • Docker Engine：

    • 负责网络 / volume / CLI
    • 把"Docker 语义"翻译成 containerd 能懂的请求

  • containerd：

    • 只负责"跑容器"
    • 不知道什么是 docker run

📌 在 containerd 看来：Docker 只是一个客户端

五、Kubernetes 是怎么"接入我"的？

Kubernetes → containerd

kubelet
   ↓
CRI (gRPC)
   ↓
containerd CRI Plugin
   ↓
containerd Core

特点

  • kubelet：

    • 说的是 Pod / Sandbox

  • CRI Plugin：

    • 把 Pod 语义翻译成 containerd 操作

  • containerd：

    • 完全不关心这是 Pod 还是 Container

📌 CRI Plugin = Kubernetes 的"翻译官"

六、containerd 内部如何"统一执行"？

不管来源是谁，containerd 最终都做这几步：

1. 拉 OCI 镜像
2. 准备 rootfs(snapshot)
3. 生成 OCI config.json
4. 调用 runc
5. 管理进程生命周期

👉 核心执行路径完全一致

              ┌─────────────┐
              │ Docker      │
              │ Engine      │
              └──────┬──────┘
                     │
              ┌──────▼──────┐

              │ gRPC API    │       (via CRI)
              └──────┬──────┘

八、为什么 docker ps 和 kubectl get pods 互相看不到？

从 containerd 的角度：

"你们是不同的客户端，我给你们分了不同的 namespace。"

containerd namespaces

  • Docker 用：moby
  • Kubernetes 用：k8s.io

所以：

ctr -n moby containers list     # Docker
ctr -n k8s.io containers list   # Kubernetes

📌 不是看不到，是你没进对 namespace

九、containerd 眼中的 CRI 和 Docker

上游containerd 看到的
Docker Engine普通 client
kubelet通过 CRI plugin 的 client
runc下游 runtime

containerd 没有"偏爱"

十、终极一句话总结(containerd 视角)

我(containerd)是中立的执行引擎： Docker 和 Kubernetes 只是用不同语言跟我说话， 但我干的活始终是同一套。

📜第五部分

用 ctr 实际直接跑一个 nginx(绕过 Docker / K8s)来看 containerd 真正的运行路径 (貌似ctr很少使用)

ctr 是 containerd 的底层调试工具，它几乎不用在生产或者日常开发中，因为大部分人都用 Docker 或 Kubernetes 来间接调用 containerd。 但是用 ctr 直接跑一个 nginx，可以让你看到 containerd 真正的运行路径，完全绕过 Docker / K8s，理解清楚"上游不同但下游一致"的本质。

一、前提条件

  1. 你的系统里已经安装 containerd 并在运行。
  2. root 权限或用 sudo。
  3. 你知道 containerd 的命名空间，默认：
  • Docker 用 moby
  • Kubernetes 用 k8s.io
  • 你可以用 default 或自定义

二、查看 containerd 镜像

先看看 containerd 已经有的镜像：

sudo ctr images list

如果没有 nginx 镜像，可以 pull：

sudo ctr images pull docker.io/library/nginx:latest

containerd 直接支持 Docker Hub 镜像，因为它遵循 OCI Image Spec。

三、用 ctr 运行 nginx

1️⃣ 创建容器(containerd 里叫 task)

sudo ctr run -d --rm docker.io/library/nginx:latest nginx-test

解释：

  • -d → 后台运行
  • --rm → 容器退出后自动删除
  • docker.io/library/nginx:latest → 镜像
  • nginx-test → 容器名字

2️⃣ 查看正在运行的容器

sudo ctr tasks list

你会看到类似：

nginx-test    RUNNING    <pid>

3️⃣ 查看日志

containerd 本身不提供日志命令，但你可以 attach：

sudo ctr tasks attach nginx-test

或查看 runc 进程：

ps aux | grep nginx

4️⃣ 停止容器

sudo ctr tasks kill nginx-test

四、和 docker run / kubectl run 对比

docker run nginxkubectl run nginxctr run nginx
镜像来源Docker pullK8s 调 containerd pullcontainerd pull
上游指令Docker CLI → dockerdkubelet → CRI → containerdctr CLI
containerd间接间接直接
OCI Runtimeruncruncrunc
Linux 进程nginxnginxnginx
查看命令docker ps / logskubectl get pods / logsctr tasks list / attach

✅ 核心区别就是**"containerd 的上游是谁"**。

五、为什么平时很少直接用 ctr？

  1. 命令复杂、不友好
  • 每个操作都要显式指定 namespace、task 名
  1. 功能太底层
  • Docker / K8s 都在 ctr 之上封装了易用 API
  1. 生产环境不推荐
  • 直接操作容易破坏容器状态，尤其是 K8s 里面

ctr 更多是调试工具 / 学习工具，实际运行容器还是用 Docker / K8s。

六、总结

  1. ctr 可以直接运行容器，绕过 Docker / K8s
  2. containerd 对任何上游都是中立的
  3. Docker / K8s 只是不同的上游客户端
  4. 底层执行(runc + Linux kernel)完全一致

📜第六部分

containerd需要单独安装吗

一、containerd 安装情况

  1. Docker 自带 containerd
  • Docker Engine 内部本身就带了 containerd(daemon)
  • 安装 Docker = 自动安装 containerd
  • 版本可能会随 Docker 升级而升级

✅ 这种情况下，普通使用 Docker 的用户不需要单独安装 containerd

  1. Kubernetes / CRI-O 需要 containerd
  • 如果你不打算用 Docker，而是直接用 containerd 或 CRI-O 作为 Kubernetes runtime
  • 就需要单独安装 containerd

举例：K8s 官方推荐从 v1.24 开始直接用 containerd，而不是 Docker Engine

  • 在 Ubuntu 上，可以通过 apt install containerd 或官方二进制安装
  • 安装后要配置 containerd.service 并启动

二、安装 vs 使用的区别

场景是否需要单独安装 containerd
只用 Docker run / build / push❌ 不需要
Kubernetes 直接用 containerd 或 CRI-O✅ 需要
学习 / 调试 ctr 命令✅ 需要(或者使用 Docker 内部的 containerd，但要找路径)

三、实际操作建议

  1. 本地开发 / 学习 Docker
  • 安装 Docker 就够了
  • containerd 自动随 Docker 启动
  1. K8s 节点 / CRI-O / containerd runtime
  • 单独安装 containerd
  • 配置 systemd 启动
  • 配置命名空间(比如 k8s.io)
  1. 调试 / ctr 实验
  • 可以直接用 Docker 内置的 containerd
  • 但如果要和 K8s 分开测试，建议单独安装

四、总结

containerd 本身就是一个独立的容器运行时，但 Docker 自带了它，所以普通 Docker 用户无需单独安装。 只有在 Kubernetes 或者想直接用 ctr 时，才需要单独安装 containerd。

📜第七部分

K8s 官方推荐从 v1.24 开始直接用 containerd，而不是 Docker Engine， 如果使用containerd 需要怎么配置，如果继续使用Docker Engine 怎么配置

一、K8s 官方推荐背景

  1. Kubernetes v1.24+ 开始 废弃对 Docker Engine 直接作为 runtime 的支持(叫做 dockershim)
  2. 原因：Docker Engine 包含太多和 K8s 无关的功能(构建、CLI、Docker Swarm 等)，维护复杂
  3. 目标：只保留 CRI 接口
  • containerd、CRI-O 原生实现 CRI
  • Docker Engine 必须通过 dockershim 才能支持(已废弃)

✅ 所以官方推荐直接使用 containerd 或 CRI-O。

二、containerd 作为 Kubernetes runtime 的配置

1️⃣ 安装 containerd

# Ubuntu 示例
sudo apt update
sudo apt install -y containerd

或者从官方 release 下载二进制：

wget https://github.com/containerd/containerd/releases/download/v1.8.8/containerd-1.8.8-linux-amd64.tar.gz
sudo tar Cxvf /usr/local containerd-1.8.8-linux-amd64.tar.gz

2️⃣ 配置 containerd

生成默认配置文件：

sudo mkdir -p /etc/containerd
sudo containerd config default | sudo tee /etc/containerd/config.toml

关键点：

  • 确保使用 systemd cgroup driver(与 kubelet 匹配)：
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
  SystemdCgroup = true
  • 命名空间(K8s 默认 k8s.io)：

containerd 默认支持多命名空间，无需额外配置，K8s kubelet 会指定 k8s.io。

3️⃣ 启动 containerd

sudo systemctl enable --now containerd
sudo systemctl status containerd

4️⃣ 配置 kubelet 使用 containerd

  • kubelet 配置文件（/var/lib/kubelet/kubeadm-flags.env 或 systemd drop-in）：
KUBELET_EXTRA_ARGS=--container-runtime=remote --container-runtime-endpoint=unix:///run/containerd/containerd.sock --runtime-request-timeout=15m
  • kubeadm 初始化或加入集群：
sudo kubeadm init --cri-socket /run/containerd/containerd.sock

✅ 这样 kubelet 就直接通过 CRI 与 containerd 通信。

三、Docker Engine 作为 Kubernetes runtime

仍然可以用，但有几个限制

  1. Kubernetes v1.24+ dockershim 已废弃
  • kubelet 不再直接支持 Docker Engine
  • 官方不再维护
  1. 如果你想用 Docker Engine，需要安装 cri-dockerd（第三方 shim）：
# Ubuntu 示例
sudo apt install -y golang-go git
git clone https://github.com/Mirantis/cri-dockerd.git
cd cri-dockerd
make
sudo make install
sudo systemctl enable --now cri-docker.service
  1. kubelet 指向 cri-dockerd socket：
--container-runtime=remote \
--container-runtime-endpoint=unix:///var/run/cri-dockerd.sock
  1. ⚠️ 这是过渡方案，不推荐生产使用

四、对比表格

特性containerdDocker Engine (via cri-dockerd)
官方推荐✅❌（废弃 dockershim）
安装复杂度低中（需要 cri-dockerd）
CRI 支持原生shim 实现
功能只跑容器包含构建、CLI、Swarm（K8s 不需要）
升级维护官方支持第三方维护
kubeadm 初始化--cri-socket=/run/containerd/containerd.sock--cri-socket=/var/run/cri-dockerd.sock

结论:

  • Kubernetes 使用 Docker Engine = 间接调用 containerd（通过 Docker Engine → containerd → OCI Runtime → Linux）
  • Kubernetes 直接使用 containerd = 直接调用 containerd → OCI Runtime → Linux

路径:

  • 使用 Docker Engine：
    kubelet → (dockershim / cri-dockerd) → Docker Engine → containerd → runc → Linux
  • 使用 containerd：
    kubelet → containerd → runc → Linux

核心差异:

  • Docker Engine 多了一层 → “绕了一圈”
  • 直接使用 containerd → 更轻量、官方推荐、少 shim 层
  • 下游执行完全一致（runc + Linux 内核）

五、CRI-O 作为 Kubernetes runtime (补充一个)

1.Kubernetes 使用 CRI-O = 直接通过 CRI 调用 CRI-O → OCI Runtime → Linux 和直接使用 containerd 类似，只是底层 runtime 是 CRI-O 而不是 containerd

2.只跑 Pod 容器

3.CRI-O 是专门为 K8s 设计的容器运行时(更轻量), OCI Runtime（runc/crun）与 containerd 一样，真正创建 Linux 进程

六、总结

  1. 官方推荐 containerd 或 CRI-O
  • 原生 CRI
  • 稳定轻量
  • 和 K8s 最新版本兼容
  1. Docker Engine 仍然可以用
  • 但需要 cri-dockerd shim
  • 不推荐生产环境
  • 只是“过渡方案”
  1. 本质区别
  • containerd：K8s 原生 runtime
  • Docker Engine：额外一层 shim 才能服务 K8s
